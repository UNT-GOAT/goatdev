Config & Endpoints

Architecture
All frontend routing is driven by /js/config.js, generated during CI/CD from GitHub secrets. Auth logic lives in /js/auth.js. Both are served from S3 via CloudFront.
CONFIG.AUTH_BASE — EC2 via ALB. Auth endpoints: /auth/login, /auth/refresh, /auth/verify, /auth/me, /auth/users
CONFIG.DB_BASE — EC2 via ALB through db-proxy. Data endpoints: /db/goats, /db/providers, /db/grading, etc.
CONFIG.API_BASE — Pi via Tailscale. Live endpoints: /api/prod/*, /api/viewfocus/*, /api/heater/*, /api/capture/*
Auth and DB never go through the Pi. The Pi is only for live camera/grading workflows. 
const CONFIG = {
  AUTH_BASE: "https://api.herd-sync.com",      // EC2 via ALB
  DB_BASE:   "https://api.herd-sync.com/db",   // EC2 via ALB → db-proxy
  API_BASE:  "https://pi.herd-sync.com",       // Pi via Tailscale
};

Auth Gate Pattern

Security
Every page starts hidden. No content renders until HerdAuth.requireAuth() resolves.
HerdAuth.fetch(url, opts) — drop-in for fetch(). Attaches Bearer token, auto-refreshes on 401.
HerdAuth.getAccessToken() — raw JWT for ?token= URLs (img tags).
HerdAuth.getUser() — cached {id, username, role}.
HerdAuth.logout() — clears tokens, revokes refresh server-side.
Never show page content before requireAuth() resolves. Never hand-roll token attachment for normal API requests. 
function showApp(user) {
  document.getElementById("app").style.display = "block";
}
// Must be LAST line in every page
HerdAuth.requireAuth().then((user) => showApp(user));

Image Auth (?token= pattern)

Security
<img> tags cannot set HTTP headers. JWT is passed as a query parameter instead. The Pi's Caddy forward_auth extracts it from X-Forwarded-Uri. Tokens auto-refresh every ~14 min via auth.js which updates all img[src*="token="] elements. 
img.src = CONFIG.API_BASE + "/api/viewfocus/stream/side"
  + "?token=" + HerdAuth.getAccessToken();

Grading & Confirmation Flow

Feature
Grading is a multi-step process. The CV model is not perfect — operators must review results before they hit the database.
1. Operator enters serial_id + live_weight, clicks Grade
2. Pi captures images → sends to EC2 → EC2 returns grade + measurements + debug images
3. Confirmation panel shows results + debug overlays for review
4. Operator clicks Confirm → frontend POSTs to CONFIG.DB_BASE + "/grading"
5. Or clicks Reject → result discarded, no DB write
EC2 never auto-saves to the database. The frontend owns the confirm/reject decision. This is intentional. 
// On confirm — save to DB through db-proxy
const resp = await HerdAuth.fetch(CONFIG.DB_BASE + "/grading", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify(pendingResult),
});
// On reject — just clear the panel, nothing saved

Pi Route Surface

Architecture
All live operational flows go through Pi routes, never directly to EC2. 
/api/viewfocus/stream/{cam}        MJPEG live stream
/api/viewfocus/settings            Camera focus settings
/api/prod/grade                    Live capture → grade
/api/prod/grade/test               Upload images → grade (no cameras)
/api/prod/status                   Poll grading progress
/api/prod/debug/{serial_id}/{view} Debug overlay images
/api/heater/status                 Heater dashboard data
/api/capture/record                Training capture

DB Routes (via db-proxy)

Architecture
All database access goes through CONFIG.DB_BASE which hits the auth-gated db-proxy. The proxy validates the JWT, then forwards to the internal db service. Only whitelisted prefixes are allowed. 
GET    /goats                List all goats
GET    /goats/{serial_id}    Get one goat
POST   /goats                Create goat (serial_id auto-assigned)
PUT    /goats/{serial_id}    Update goat

GET    /grading              Recent grade results
GET    /grading/{serial_id}  Grade history for a goat
POST   /grading              Save a confirmed grade result

GET    /providers             List providers
GET    /animals/next          Preview next serial_id

Same pattern for /chickens, /lambs.

Security Model

Security
Two independent auth layers protect the system.
User-level: Browser → Bearer JWT → Caddy forward_auth (Pi) or db-proxy verify (EC2)
Service-level: Pi → EC2 via X-API-Key header (shared secret in .env)
EC2 API middleware fails closed: no key configured = all non-health requests rejected. Pi↔EC2 traffic goes over Tailscale (WireGuard encrypted). 
Browser → Pi (Caddy forward_auth) → EC2 (X-API-Key)
Browser → ALB → auth service (login/refresh/verify)
Browser → ALB → db-proxy (JWT verify) → db service

New Page Checklist

Pattern
Every new page must follow these rules:
• Include /js/config.js and /js/auth.js
• Gate with HerdAuth.requireAuth().then(showApp) as last line
• Use HerdAuth.fetch() for all authenticated requests
• Use ?token= for img/stream src attributes
• Use CONFIG.API_BASE for Pi, CONFIG.AUTH_BASE for auth, CONFIG.DB_BASE for data
• Role checks are UI-only — backend enforces actual permissions 
<script src="/js/config.js"></script>
<script src="/js/auth.js"></script>
<script>
  function showApp(user) { /* render page */ }
  HerdAuth.requireAuth().then(showApp);
</script>
