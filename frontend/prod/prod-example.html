System Overview

Architecture
HerdSync is a frontend-to-Pi-to-backend system. A frontend should treat the Pi-facing API surface as the main integration layer for live production workflows, while auth and database services remain backend-owned.
Core layers:
• Frontend — authenticated browser UI
• Pi routes — live camera access and grading orchestration
• Auth service — login, session validation, users
• DB service — animal/provider/grading data access
• EC2 grading service — measurement and grade pipeline
Frontend rule: never connect directly to the database, never own grading logic, and never assume camera access exists outside the Pi surface routes. 
Frontend
  │
  ├─ Authenticated API calls
  │
  ▼
Pi-facing routes
  ├─ live camera streams
  ├─ grading orchestration
  ├─ debug image access
  │
  ▼
Backend services
  ├─ auth
  ├─ db
  └─ ec2 grading

Database and grading internals stay backend-side.

Authentication Model

Auth
Every new frontend page should be built around the HerdAuth client contract.
Required behaviors:
• Gate page rendering with HerdAuth.requireAuth()
• Use HerdAuth.fetch() for authenticated API requests
• Read current identity from the resolved user object or HerdAuth.getUser()
• Use HerdAuth.logout() for session exit
Frontend rule: do not hand-roll token attachment or refresh behavior for normal API requests. Treat HerdAuth.fetch() as the default transport. 
function showApp(user) {
  document.getElementById("app").style.display = "block";
}

HerdAuth.requireAuth().then((user) => showApp(user));

Identity and Role Usage

Pattern
The authenticated user object is the frontend source of truth for identity-aware UI.
Expected user shape:
• id
• username
• role
Role checks in the frontend are primarily for navigation and UI shaping. They do not replace backend authorization.
Frontend rule: use role checks to show or hide admin/operator experiences, but assume backend routes still own final permission enforcement. 
if (user.role === "admin") {
  // show admin navigation
}

Image and Stream Authentication

Security
Stream and debug image routes are consumed through <img> tags, which cannot attach Bearer headers. Because of that, image-based routes use a token-in-URL pattern.
Pattern:
• normal API requests → HerdAuth.fetch()
• image/stream routes → append ?token=<jwt>
Frontend rule: any new UI that renders live streams or protected images must use the query-token pattern rather than trying to fetch them as normal authenticated JSON endpoints. 
img.src =
  CONFIG.API_BASE +
  "/api/viewfocus/stream/side?token=" +
  HerdAuth.getAccessToken();

Pi Integration Surface

Architecture
A new frontend integrating production functionality should treat the Pi as the entry point for all live grading interactions.
Main Pi-facing route groups:
• /api/viewfocus/* — live camera stream access
• /api/prod/* — live capture, grading orchestration, status, debug artifacts
Frontend rule: for live operational flows, build against Pi routes, not directly against EC2 grading internals. 
Pi route groups used by a production frontend:

/api/viewfocus/stream/{cam}
/api/prod/grade
/api/prod/grade/test
/api/prod/status
/api/prod/debug/{serial_id}/{view}

Live Camera Integration

Feature
The live production UI is built around three camera views: side, top, and front. These should be treated as fixed named views across the system.
Expected frontend behavior:
• render all three streams when available
• keep view names consistent across live preview, upload, debug, and measurement displays
• assume downstream grading and debug logic also use the same three-view naming contract 
["side", "top", "front"].forEach((cam) => {
  document.getElementById("stream-" + cam).src =
    CONFIG.API_BASE +
    "/api/viewfocus/stream/" +
    cam +
    "?token=" +
    HerdAuth.getAccessToken();
});

Grading Workflow Contract

Feature
A production frontend should support two grading entry modes:
1. Live capture mode
Uses /api/prod/grade and lets the Pi capture images from connected cameras.

2. Manual upload mode
Uses /api/prod/grade/test and submits uploaded images directly for grading without requiring live camera capture.
Both modes are built around a shared grading input shape centered on animal identity and live weight.
Frontend rule: treat upload mode as a testing or fallback path, and live capture mode as the primary production workflow. 
Shared grading concepts:
  serial_id
  live_weight
  side image
  top image
  front image

Status Polling Model

Pattern
Live grading is asynchronous from the frontend point of view. After starting a grade job, the UI should poll the status endpoint until the job becomes inactive and returns a final result or error.
Status concepts surfaced to frontend:
• active / inactive state
• progress stage
• final result payload
• last error payload
Typical progress stages: starting, capturing, grading, archiving.
Frontend rule: build the UI around progress states, not around assumptions of instant completion. 
const resp = await HerdAuth.fetch(CONFIG.API_BASE + "/api/prod/status");
const data = await resp.json();

Grading Result Shape

Feature
Frontends should expect a grading result to include both the final business result and diagnostic metadata.
Important result fields used by this page:
• grade
• all_views_successful
• timing
• measurements
• warnings
This lets the frontend support both operator-friendly output and deeper troubleshooting views. 
Grade result should be treated as:
  outcome
  measurement payload
  timing metadata
  warning list
  view success metadata

Debug Image Contract

Feature
Debug imagery is a first-class part of the grading workflow. A new frontend should be able to request view-specific debug overlays after a completed run.
Route pattern:
/api/prod/debug/{serial_id}/{view}
These images are useful for reviewing measurement overlays and confirming view-level grading behavior.
Frontend rule: keep the same canonical view names for debug rendering: side, top, front. 
const url =
  CONFIG.API_BASE +
  "/api/prod/debug/" +
  serialId +
  "/" +
  view +
  "?token=" +
  HerdAuth.getAccessToken();

Database Integration Model

Architecture
A frontend should integrate with animal and grading data through backend domain routes, not table-level assumptions.
Domain resources already exposed to frontend:
• /goats
• /chickens
• /lambs
• /providers
• /grading
Frontend rule: think in terms of domain resources, not joins, raw SQL, or direct table ownership. 
const endpoint =
  "/" + table + (table === "grading" ? "?limit=200" : "");
const data = await dbFetch(endpoint);

Backend Domain Model

Pattern
The DB/backend layer is organized around domain route groups rather than a single generic data handler.
Current route groups:
• animals — shared animal operations
• animal_factory — common creation/routing logic
• goats — goat-specific data
• chickens — chicken-specific data
• lambs — lamb-specific data
• grading — grading records and outcomes
• providers — provider/source records
This structure is the conceptual model a frontend should mirror when building navigation, filters, views, or forms. 
db/routes/
  animal_factory.py
  animals.py
  goats.py
  chickens.py
  lambs.py
  grading.py
  providers.py

Schema Awareness

Architecture
Frontends should be schema-aware enough to present meaningful records, but not schema-dependent in a way that bypasses backend contracts.
The schema diagram is useful for understanding relationships between animal records, providers, and grading results, but the frontend should still consume backend-shaped JSON rather than re-implementing relational logic itself.
Frontend rule: use schema knowledge for UI design and mental model only. Let backend routes own relational behavior. 
Use schema knowledge to understand:
  which resources relate
  which details belong in record views
  which cross-links make sense in the UI

Do not use schema knowledge to bypass backend APIs.

Password Management Flow

Auth
User account settings belong to the auth service boundary. Password-change UX should preserve the currently authenticated session while validating the user's current password.
Current pattern:
• verify current password through /auth/login
• submit the actual password update through /auth/users/{id}
This split is useful when building account settings pages in other frontends too. 
Account settings frontend flow:
  verify identity with current credentials
  then submit authenticated account update

New Frontend Build Checklist

Pattern
A new frontend integrating the full system should preserve these invariants:
• Gate all pages with HerdAuth.requireAuth()
• Use HerdAuth.fetch() for authenticated JSON requests
• Use query-token auth for image/stream routes
• Integrate live workflows through Pi routes
• Treat grading as asynchronous and poll status
• Use canonical camera/view names: side, top, front
• Consume backend domain resources for records and grading history
• Keep DB, auth, and grading logic server-owned 
Build a new frontend like this:
  auth-gated
  Pi-integrated for live workflows
  backend-driven for records
  async-aware for grading
  consistent in view naming
  diagnostic-friendly
